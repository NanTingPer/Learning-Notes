# 装饰器模式

装饰器模式就是将一个原版的服务，进行二次封装。

如，我现在需要将一个查询服务应用缓存，而原服务是直接从数据库查询的。我想原服务支持缓存查询，那么就可以使用装饰器模式，对原服务进行装饰。

## 原服务

原服务中有多种方法，包含了完全的CRUD功能。并继承了` BaseRepository<PostInfo, string>(context)`抽象，现在我们需要对他进行装饰。

```cs
public class PostsService(BlogContext context): BaseRepository<PostInfo, string>(context)
```

## 抽象

装饰器模式最依赖的模式便是抽象，将原服务抽象为多个接口以进行组合，最终进行封装，以支持原服务的完整功能。

以上面的`PostsService`为例，我们可以抽象一个`IPostsService`，而此接口将会要求实现`PostsService`全部的公共方法，以支持完全装饰。

如果要进一步抽象，还可以抽象一个`IPostsQueryService`和`IBaseRepository`，前者为服务的查询接口，后者为对`BaseRepository`的抽象。最终文件大概会如下。

```cs
public interface IBaseRepository<TModel, TKey>
public interface IPostsQueryService
public interface IPostsService: IPostsQueryService, IBaseRepository<PostInfo, string>;
public class PostsService(BlogContext context) : BaseRepository<PostInfo, string>(context), IPostService
```

这样，我们就对`PostsService`完整的抽象出来了，如果要对他进行装饰，只需要创建实现`IPostsService`的类型，并要求传入`PostsService`，在实现中调用原服务的方法，或直接对原服务的内容进行返回，也可以对原服务的内容进行加工 或 使用缓存中的数据。

```cs
// 注入具体实现 PostsService，而不是 IPostService，
// 是为了让后台刷新缓存的服务可以单独解析出不带缓存的原始服务。
public class PostsCacheService(PostsService service, IMemoryCache cache) : IPostService
{
    // 可以先对cache进行尝试获取，如果没有 则从原始服务返回并写缓存
    public List<PostInfo> QueryByLast(int limit, int page) { ... }
}
```

## 依赖注入

```cs
// 注册原始服务
services.AddScoped<PostsService>();

// 注册 IPostService，指向装饰器
services.AddScoped<IPostService>(sp =>
{
    var inner = sp.GetRequiredService<PostsService>();
    var cache = sp.GetRequiredService<IMemoryCache>();
    return new PostsCacheService(inner, cache);
});
```
