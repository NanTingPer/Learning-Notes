编写完MCP服务后，我对AI的看法。
	MCP[服务的文档](https://modelcontextprotocol.io/docs/getting-started/intro)。就接入小智的MCP服务而言。他给我提供了一个`wss`连接，我需要使用`ClientWebSocket`去进行链接。当然，第一反应当然是使用`Postman`去请求这个`websocket`，一连接上去他就发了个初始化请求。

​	我对`MCP`协议的认知一片空白。本来不想重复造轮子，就想起了微软官方的`AI`扩展包，或者说`MCP C# SDK`，但是那个包还在预览阶段，虽然这个不是主要问题。主要问题是，我作为服务器需要主动去连接客户端，寻找各种解决方案都是使用`Http反向代理`，将`websocket`请求转移到`http api 终结点`上。

​	翻阅`SKD / Nuget`包文档，其实是根本看不下去的。虽然阅读文档是程序员的一种硬实力，但是能阅读下去的前提是，找到你需要的部分。

​	我本身对于`.NET System.Net`包不是很熟悉，`WebSocket`是抽象类，不能创建实例，之前使用过几次`ClientWebSocket`但太久没接触也就忘了有这个类，但是我第一反应不是去看文档中对于`WebSocket`的实现，而是去询问`DeepSeek`，问他 如何连接`wss`，他有给我示例代码，我也只从示例代码中摘取了一个类型 `ClientWebSocket` 然后就闷头干去了。

​	连上后就是翻阅MCP官方文档，讲的很详细，也是我想要的内容，所以文档我也理所当然的看下去了。但是对于JsonObject映射到C#强类型，做这种模型类是非常枯燥的，我就索性把官方给的结构丢给DeepSeek了。这也是我认为，AI在真正的项目中能胜任的事。

​	我不认为AI能真正很好的完成一个项目，他只能完成小模块。对于过渡神圣或许过渡低估，我会低估也会神圣。我很依赖AI也很不依赖AI，我觉得他在对文档进行总结 并输出我想看到的结构的时候很有用。但是在作出编码建议的时，我又不觉得有用了。

​	这跟我写MCP服务有什么关联？我在编写的过程中，结合文档向DeepSeek询问，这样大大减少了我是思考，这是我认为必须要考虑的一个问题。形成路径依赖后就会越来越不喜欢思考，什么东西都丢给AI，让AI干，但是使用AI本身是要付出成本的，去干的这件事或许你消费了东西，但是得到的是无意义的内容。但是又因为你过渡依赖了，导致你根本不想去思考，就一种跟他耗着，想着 都思考那么久了，总该行了。可有这个时间，正常来说问题应该已经解决了。
